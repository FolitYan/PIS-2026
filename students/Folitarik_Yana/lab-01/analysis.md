# Анализ границ ответственности

**Система:** Микро-твиттер 🐦  
**Сценарий:** Публикация нового поста

---

## 1. Транзакционные границы

### 1.1. Где начинается и заканчивается транзакция?

**Начало транзакции:** Автор нажимает кнопку «Опубликовать».

**Конец транзакции (успешный сценарий):**
- **Основная транзакция БД:** Пост сохранён + счётчик постов автора увеличен.
- **Побочные эффекты (асинхронные):** Обновление лент подписчиков, отправка уведомлений (если требуется).

**Важно:** У нас одна основная транзакция, которая гарантирует сохранность поста. Всё остальное может быть выполнено асинхронно, без угрозы потери данных.

### 1.2. Какие операции должны быть атомарными?

**Атомарные операции (в одной БД‑транзакции):**
1. Валидация входных данных (длина текста, наличие запрещённых символов).
2. Вызов сервиса модерации (синхронно, но результат может повлиять на решение о сохранении).
3. Создание записи поста в таблице `posts`.
4. Увеличение счётчика постов пользователя в таблице `users` (поле `posts_count`).
5. Генерация уникального ID поста.

**Обоснование:** Пост без увеличения счётчика нарушает целостность профиля. Если модерация отклоняет пост — транзакция не выполняется.

**Пример сбоя:**
- Если пост сохранён, но увеличение счётчика не удалось (например, из‑за блокировки) → ROLLBACK всей транзакции. Иначе у нас будет «пост‑призрак», не учитываемый в статистике.

### 1.3. Какие операции могут быть асинхронными?

| Операция | Обоснование |
|----------|-------------|
| Обновление лент подписчиков (ленты новостей) | Допустима eventual consistency. Подписчики увидят пост чуть позже, это не критично. |
| Отправка push‑уведомлений подписчикам | Не влияет на факт публикации. Если уведомление не доставлено, пост всё равно существует. |
| Запись в историю активности / аудит‑лог | Можно сохранить асинхронно, ошибка не нарушает основной функционал. |
| Обновление кэша (например, трендов) | Данные можно пересчитать позже. |
| Индексация для поиска | Допустима задержка. |

**Принцип:** Если операция не нарушает инварианты «пост должен быть сохранён и виден автору» — она может быть асинхронной.

## 2. Таблица транзакционных границ

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry‑стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| **Валидация длины текста** | Синхронная | Нет (просто ошибка) | N/A | Да |
| **Вызов Moderation Service** | Синхронная | Да (ROLLBACK транзакции, если модерация не прошла) | 3 попытки с задержкой 200ms | Да (ключ по тексту + автор) |
| **Создание записи поста в БД** | Синхронная | Да (ROLLBACK) | Нет (контролируется БД) | Да (проверка дублей по `idempotency_key`) |
| **Увеличение счётчика постов автора** | Синхронная | Да (в той же транзакции) | Нет | Да (оптимистичная блокировка с `version`) |
| **Генерация уникального post_id** | Синхронная | Нет (детерминировано) | N/A | Да (UUID или sequence) |
| **Публикация доменного события (PostCreated)** | Синхронная (через transactional outbox) | Да (часть транзакции) | Нет | Да (уникальный event_id) |
| **Обновление лент подписчиков** | Асинхронная (через очередь) | Нет (пост уже сохранён) | 5 попыток с exponential backoff | Да (дедупликация по `post_id + subscriber_id`) |
| **Отправка push‑уведомлений** | Асинхронная (через очередь) | Нет | 5 попыток | Да |
| **Индексация для поиска** | Асинхронная | Нет | 3 попытки | Да |

## 3. Обработка отказов внешних сервисов

### 3.1. Moderation Service недоступен

**Проблема:** Сервис модерации не отвечает (таймаут, 5xx).

**Стратегия:**
1. **Таймаут:** 3 секунды.
2. **Действие:** Если модерация не отвечает, транзакция **не выполняется** (пост не публикуется).
3. **Retry:** 3 попытки с задержкой 200ms, 500ms, 1s.
4. **После неудачи:** Возвращаем пользователю ошибку «Сервис модерации временно недоступен, попробуйте позже».
5. **Альтернатива:** Можно сохранить пост в статусе `PENDING_MODERATION` и опубликовать позже, но это усложнит логику. В упрощённом варианте — блокируем публикацию.

**Обоснование:** Модерация критична для безопасности платформы. Лучше отказать в публикации, чем пропустить нежелательный контент.

### 3.2. База данных недоступна

**Проблема:** PostgreSQL не отвечает (connection refused, timeout).

**Стратегия:**
1. **Таймаут:** 5 секунд на соединение, 10 секунд на запрос.
2. **Транзакция:** ROLLBACK (пост не создаётся).
3. **Retry:** 3 попытки на уровне приложения (1s, 2s, 4s) → если все неудачны, возвращаем 503.
4. **Offline‑режим:** Если пользователь вводит пост при отсутствии интернета, сохраняем черновик локально (LocalStorage/IndexedDB) и синхронизируем при восстановлении связи.
5. **Уведомление пользователя:** «Не удалось опубликовать пост. Он сохранён как черновик и будет отправлен, когда соединение восстановится».

**Обоснование:** Без БД пост не может быть сохранён. Для мобильных сценариев критичен offline‑режим.

### 3.3. Конкурентное обновление счётчика постов

**Проблема:** Два поста публикуются одновременно, и счётчик может быть увеличен некорректно.

**Стратегия:**
1. Используем **оптимистичную блокировку** с полем `version` в таблице пользователей.
   ```sql
   UPDATE users 
   SET posts_count = posts_count + 1, version = version + 1 
   WHERE id = ? AND version = ?;
   ```
2. Если обновление не затронуло строк (версия изменилась), транзакция откатывается, и клиент получает ошибку «Повторите попытку».
3. На клиенте можно автоматически повторить запрос (с новым `idempotency_key`).

**Обоснование:** Оптимистичная блокировка проще пессимистичной и достаточно эффективна при низкой частоте конфликтов.

## 4. Идемпотентность

### 4.1. Проблема: Повторное нажатие кнопки «Опубликовать»

**Сценарий:** Автор дважды кликнул из‑за медленной сети или собственной невнимательности.

**Решение:**
1. **Frontend:** Блокируем кнопку после первого нажатия (disable + спиннер).
2. **Backend:** Принимаем `idempotency_key` в запросе (например, UUID, сгенерированный на клиенте).
   ```json
   {
     "text": "Мой первый пост!",
     "idempotency_key": "client-abc-123"
   }
   ```
3. **Логика:**
   - Проверяем таблицу `idempotent_requests` по ключу.
   - Если ключ уже есть → возвращаем ранее сохранённый ответ (post_id, статус).
   - Если нет → выполняем транзакцию и сохраняем результат в `idempotent_requests`.
4. **TTL ключей:** 24 часа (достаточно для исключения дублей при повторных отправках).

**Пример таблицы:**
```sql
CREATE TABLE idempotent_requests (
  idempotency_key VARCHAR(128) PRIMARY KEY,
  response_body JSONB NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);
-- Очистка старых записей ежедневно
```

### 4.2. Дубликаты из‑за retry при отправке в очередь

**Сценарий:** Worker, обновляющий ленты подписчиков, получил задание дважды.

**Решение:**
- Используем уникальный индекс в таблице задач:
  ```sql
  CREATE UNIQUE INDEX idx_feed_update_dedup 
  ON feed_update_queue(post_id, subscriber_id);
  ```
- При конфликте — игнорируем (`ON CONFLICT DO NOTHING`).

## 6. Метрики и мониторинг

| Метрика | Цель | Алерт при |
|---------|------|-----------|
| `post_creation_latency_p95` | < 500 мс | > 2 с |
| `post_creation_success_rate` | > 99% | < 98% |
| `moderation_service_timeout_rate` | < 1% | > 5% |
| `db_connection_pool_usage` | < 80% | > 90% |
| `feed_update_queue_size` | < 1000 | > 5000 |
| `idempotency_key_collisions` | 0 | > 10/час |
| `offline_sync_pending_count` | 0 | > 100 |

**Логирование:**
- **INFO:** Успешная публикация поста.
- **WARNING:** Moderation Service timeout (перешли на retry).
- **WARNING:** Конфликт версий при обновлении счётчика (требуется повтор).
- **ERROR:** БД недоступна, откат транзакции.
- **ERROR:** Moderation Service вернул ошибку (пост отклонён).

## 7. Выводы

### Ключевые решения:

1. ✅ **Одна транзакция:** Пост + счётчик постов — атомарно.
2. ✅ **Синхронная модерация:** Критична для безопасности, блокирует публикацию при сбое.
3. ✅ **Асинхронное обновление лент:** Допустима eventual consistency.
4. ✅ **Идемпотентность:** `idempotency_key` защищает от дублей.
5. ✅ **Оптимистичная блокировка:** Для счётчика постов, избегаем race condition.
6. ✅ **Offline‑режим:** Черновики сохраняются локально при отсутствии соединения.
7. ✅ **Retry для внешних сервисов:** Exponential backoff для модерации и очередей.
8. ✅ **Мониторинг:** Ключевые метрики публикации и очередей.

### Транзакционные инварианты:

- **Инвариант №1:** Пост не может существовать без привязки к автору.
- **Инвариант №2:** Счётчик постов автора всегда равен количеству его постов.
- **Инвариант №3:** Один пост не может быть дважды учтён в лентах подписчиков (дедупликация).

### Компромиссы:

- **Синхронная модерация:** Увеличивает время отклика, но обеспечивает безопасность.
- **Асинхронное обновление лент:** Подписчики видят пост с задержкой, но это приемлемо.
- **Offline‑черновики:** Требуют хранения на клиенте, но улучшают UX.

### Риски:

⚠️ **Риск 1:** При массовом наплыве постов очередь обновления лент может переполниться.  
**Митигация:** Авто‑масштабирование воркеров, алерт при росте очереди.

⚠️ **Риск 2:** Если модерация станет медленной (например, из‑за ML‑модели), время публикации вырастет.  
**Митигация:** Кэширование результатов для часто встречающихся фраз, возможность асинхронной модерации для доверенных авторов.
