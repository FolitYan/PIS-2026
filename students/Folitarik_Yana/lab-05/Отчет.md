## Цель работы

Реализовать инфраструктурный слой (Infrastructure Layer) системы, включающий входящие адаптеры (REST API) и исходящие адаптеры (Persistence Repository, Event Publisher) для связи ядра системы с внешним миром.

**Вариант №20 — Post Service (Система публикаций)**  
Ядро домена: Агрегат `Post`, объекты-значения `PostContent`, `AuthorId`, перечисление `PostStatus`.

## Ход выполнения работы

### 1. Repository (PostgreSQL)

**Технологии:** Spring Data JPA, Hibernate, PostgreSQL.

В проекте реализован исходящий адаптер `PostRepositoryAdapter`, который является реализацией порта `PostRepository`. Для хранения данных используется сущность `PostJpaEntity`.

**Реализованные методы:**
- `save(Post)` — маппинг доменной модели в JPA-сущность и сохранение в базу данных.
- `findById(UUID)` — поиск записи в БД и восстановление доменного агрегата.

**Маппинг:** Для сохранения "чистоты" домена (отсутствие пустых конструкторов и сеттеров), восстановление состояния агрегата реализовано через **Java Reflection API** в адаптере.

**Схема таблицы в БД:**  
Таблица `posts` содержит колонки:  
`id` (UUID), `author_id` (VARCHAR), `content` (VARCHAR 280), `status` (VARCHAR), `likes_count` (INT).

---

### 2. REST Controller

**Эндпоинты API:**

| Метод | Path | Описание |
|-------|------|----------|
| POST | `/api/posts` | Создать новую публикацию (команда) |
| GET  | `/api/posts/{id}` | Получить данные публикации по ID (запрос) |

**Логика:** Контроллер принимает JSON, преобразует его в команду/запрос и делегирует выполнение соответствующему хэндлеру (Handler) из прикладного слоя. Возвращает статус `201 Created` при успешном создании.

### 3. Docker Compose

Для локального запуска базы данных PostgreSQL используется Docker Compose.

**`docker-compose.yml`:**
```yaml
version: '3.8'
services:
  db:
    image: postgres:15-alpine
    container_name: post_db_container
    environment:
      POSTGRES_DB: post_db
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

## Контрольные вопросы

**1. Почему Repository находится в Infrastructure, а не в Domain?**  
Домен должен содержать только чистую бизнес-логику и быть независимым от инструментов. Реализация репозитория (SQL-запросы, настройки Hibernate) — это техническая деталь. Если поместить ее в домен, мы нарушим принцип изоляции, и изменение БД заставит нас менять бизнес-код.

**2. В чём преимущество ORM над обычным SQL?**  
ORM (Object-Relational Mapping) позволяет работать с данными как с объектами, скрывая сложность SQL-запросов. Это ускоряет разработку, автоматически управляет связями объектов, защищает от SQL-инъекций и позволяет легко менять диалект БД без переписывания запросов вручную.
