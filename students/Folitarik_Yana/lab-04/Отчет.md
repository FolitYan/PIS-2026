## Цель работы

Реализовать прикладной слой (Application Layer) системы с разделением операций на команды (изменяют состояние домена) и запросы (читают данные без изменений) в соответствии с паттерном **CQRS**.

**Вариант №20 — Post Service (Система публикаций)**  
- Питч: Платформа для публикации коротких заметок и взаимодействия пользователей через лайки.  
- Ядро домена: Агрегат `Post`, объекты-значения `PostContent`, `AuthorId`, `PostStatus`.

---

## Ход выполнения работы

### 1. Команды (Commands)

В прикладном слое созданы иммутабельные объекты команд для изменения состояния системы.

**`CreatePostCommand`** — создание новой публикации.  
- Поля: `String authorId`, `String text`.  
- Валидация: проверка на `null` и пустые строки реализована в конструкторе соответствующих Value Objects в домене.  
- Файл: `application/command/CreatePostCommand.java` (реализовано как Java Record).

**Пример кода команды:**
```java
public record CreatePostCommand(String authorId, String text) {}
```

---

### 2. Command Handlers

Реализованы обработчики, инкапсулирующие логику выполнения команд.

**`CreatePostHandler`** — отвечает за создание агрегата.  
- Шаги обработки: трансформация данных команды в VO, создание агрегата `Post`, сохранение через `PostRepository`, публикация события `PostCreatedEvent`.  
- Возвращает: `UUID` (идентификатор созданного поста).  
- Файл: `application/command/handler/CreatePostHandler.java`

**Пример кода хэндлера:**
```java
@Service
public class CreatePostHandler {
    private final PostRepository repository;
    private final EventPublisher publisher;

    public CreatePostHandler(PostRepository repository, EventPublisher publisher) {
        this.repository = repository;
        this.publisher = publisher;
    }

    @Transactional
    public UUID handle(CreatePostCommand cmd) {
        // Оркестрация: Домен -> Репозиторий -> События
        Post post = Post.create(new AuthorId(cmd.authorId()), new PostContent(cmd.text()));
        repository.save(post);
        post.getEvents().forEach(publisher::publish);
        return post.getId();
    }
}
```

### 3. Queries (Запросы) и DTO

Для чтения данных реализованы запросы и упрощённые модели данных (DTO).

**Запросы:**  
- `GetPostQuery` — получение публикации по ID.  
  - Поля: `UUID postId`.  
  - Файл: `application/query/GetPostQuery.java`

**Read DTOs:**  
- `PostResponse` — модель для передачи данных клиенту.  
  - Поля: `id`, `content`, `status`, `likes`.  
  - Файл: `application/query/dto/PostResponse.java`

**Пример кода:**
```java
public record PostResponse(String id, String content, String status, int likes) {}
```

---

### 4. Query Handlers

**`GetPostHandler`** — обработчик запроса на получение данных.  
- Репозиторий: использует `PostRepository` для поиска.  
- Возвращает: `PostResponse` (преобразованный из доменной модели).  
- Файл: `application/query/handler/GetPostHandler.java`

**Пример кода:**
```java
@Service
public class GetPostHandler {
    private final PostRepository repository;

    public PostResponse handle(GetPostQuery q) {
        return repository.findById(q.postId())
            .map(p -> new PostResponse(p.getId().toString(), p.getContent().value(), 
                                      p.getStatus().name(), p.getLikesCount()))
            .orElseThrow();
    }
}
```

## Контрольные вопросы

**1. В чём разница между Command и Query?**  
Команда (Command) выражает намерение изменить состояние системы (Create, Update, Delete) и обычно возвращает только ID созданного ресурса. Запрос (Query) предназначен исключительно для чтения данных, не изменяет состояние и возвращает DTO.

**2. Почему Command Handler возвращает только ID, а не весь объект?**  
Это минимизирует передачу данных и разделяет ответственность. Клиент получает ID, чтобы при необходимости выполнить отдельный запрос (Query) для получения актуального представления объекта. Это предотвращает утечку доменной логики в слой представления.

**3. Где должна выполняться валидация?**  
Валидация формата данных (тип, обязательность) — в Команде/Handler. Валидация бизнес-инвариантов (правила системы) — строго внутри Доменной модели.

**4. Можно ли вызывать Query из Command Handler?**  
Технически возможно, но нарушает принцип чистого CQRS. Для получения данных внутри Handler лучше обращаться напрямую к репозиторию.

**5. Зачем разделять Request DTO и Command?**  
Это позволяет независимо изменять внешний API (JSON формат) и внутреннюю логику приложения. Это защищает прикладной слой от изменений в веб-фреймворках.
