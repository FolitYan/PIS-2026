# Цель работы

Заменить стандартное REST-взаимодействие между компонентами системы на высокопроизводительный протокол gRPC, реализовать типизированные контракты через Protocol Buffers и внедрить механизмы серверного стриминга.

## Вариант №20 — Post Service (gRPC Реализация)

### 1. Протофайлы (.proto)

Был разработан контракт post.proto, описывающий сервис PostGrpcService. Использование Protocol Buffers позволило определить строгую типизацию данных и автоматически сгенерировать клиентские и серверные заглушки (stubs) для Java.

### 2. gRPC Server

Реализован серверный адаптер, поддерживающий:
- Unary RPC (GetPost): Классический запрос-ответ для получения данных поста.
- Server Streaming (StreamAuthorPosts): Позволяет клиенту открыть поток и получать новые публикации автора по мере их появления без повторных запросов.

### 3. Сравнение gRPC и REST

В ходе работы было отмечено, что gRPC работает значительно быстрее за счет использования бинарного формата сериализации (Protobuf) вместо текстового (JSON) и протокола HTTP/2, который поддерживает мультиплексирование запросов в одном соединении.

## Контрольные вопросы

**В чём преимущество gRPC над REST?**  
gRPC использует HTTP/2 (бинарный протокол) против HTTP/1.1 (текстовый) у REST. Это дает меньший размер пакетов, поддержку стриминга и автоматическую генерацию кода для разных языков программирования.

**Почему Protocol Buffers быстрее JSON?**  
JSON — это текст, который нужно парсить и сопоставлять с полями. Protobuf — это бинарный формат, где данные упакованы максимально плотно, а структура (теги полей) известна заранее, что делает сериализацию почти мгновенной.

**Зачем нужен streaming в gRPC?**  
Стриминг позволяет передавать большие объемы данных по частям или поддерживать постоянное соединение для передачи обновлений в реальном времени (например, лента новостей или чат), не создавая накладных расходов на новые HTTP-запросы.
