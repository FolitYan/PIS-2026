## Цель работы
Реализовать паттерн CQRS, разделив доменную модель (Write Model) и модель представления (Read Model), обеспечив их асинхронную или событийную синхронизацию.

## Вариант №20 — Post Service (Система публикаций)
1. Write Model
Использован агрегат Post. Он отвечает за выполнение команд, проверку инвариантов и генерацию доменных событий. Данные хранятся в нормализованном виде в таблице posts.
2. Read Model
Создана специализированная таблица post_details_view. Она денормализована: данные, которые ранее требовали JOIN (например, информация об авторе), теперь хранятся в плоском виде непосредственно в записи поста. Это позволяет выполнять чтение за константное время.
3. Event-Driven Synchronization
Для синхронизации моделей реализован компонент PostProjection. Он использует механизм Spring EventListener. При возникновении события PostCreatedEvent в основном агрегате, проекция автоматически создает соответствующую запись в базе данных для чтения.
4. Оптимизация запросов
На уровне БД добавлены индексы на поля authorName и lastUpdatedAt. Это оптимизирует наиболее частые сценарии использования социальной сети: просмотр профиля конкретного пользователя и просмотр последних новостей.

## Контрольные вопросы
1. В чём разница между CQRS и CQS?
   CQS — разделение методов внутри одного класса. CQRS — разделение моделей и путей данных на уровне всей архитектуры.
2. Почему Read Model денормализованная?
   Для исключения ресурсоемких операций JOIN и подготовки данных под конкретные нужды UI.
3. Что такое Eventual Consistency?
   Временной лаг, при котором Read Model может на короткое время отставать от Write Model, но в итоге придет в соответствие.
