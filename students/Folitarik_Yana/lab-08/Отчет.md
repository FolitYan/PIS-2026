## Цель работы
Разделить монолитную систему на независимые микросервисы (Bounded Contexts), настроить межсервисное асинхронное взаимодействие через брокер сообщений (RabbitMQ) и организовать единую точку входа через API Gateway.

## Вариант №20 — Post Service (Микросервисная архитектура)

### 1. Выделение Bounded Contexts
Система была разделена на два контекста:
- **Post Service**: Ответственен за создание и хранение основного контента публикаций.
- **Interaction Service**: Ответственен за лайки, комментарии и уведомления. Каждый сервис имеет собственную изолированную базу данных.

### 2. Event Bus (RabbitMQ)
Для синхронизации данных между сервисами использован брокер сообщений RabbitMQ.
При создании поста в Post Service генерируется событие `PostCreated`, которое отправляется в Exchange.
Interaction Service подписан на очередь и получает это сообщение для создания начальных счетчиков взаимодействия.

### 3. API Gateway (Nginx)
В качестве шлюза настроен Nginx. Он принимает все внешние запросы на порту 80 и перенаправляет их:
- `/api/posts/**` → в микросервис публикаций.
- `/api/interactions/**` → в микросервис взаимодействий.

## Контрольные вопросы

**1. Что такое bounded context?**  
Это явно очерченная граница, внутри которой доменная модель имеет смысл. В микросервисах каждый сервис соответствует одному контексту.

**2. Почему микросервисы не должны делить БД?**  
Общая БД создает сильную связанность (tight coupling). Если изменится схема для одного сервиса, это может сломать другой. Микросервисы должны быть независимы.

**3. Зачем нужен Circuit Breaker?**  
Чтобы предотвратить каскадное падение всей системы. Если один сервис перегружен, Circuit Breaker "разрывает цепь", возвращая быстрый отказ или данные из кэша, не дожидаясь тайм-аута.
